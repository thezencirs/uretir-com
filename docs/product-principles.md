# Product Principles

These principles are decision tools. When delivery speed, visual polish, growth, and technical simplicity conflict, use them to make the trade-off explicit.

## 0. Pass the sustainable growth gate

Every new feature must materially improve Google traffic, session depth, returning visits, internal linking, authority, community engagement, or AI usefulness. Name the primary outcome, behavior change, and measurement before implementation. Postpone work that has no credible connection to one of these outcomes. The priority order is traffic, authority, community, then monetization.

## 1. Build for measurable value

Do not ship features for appearance, parity, or a roadmap checkbox. Define the user job, expected behavior change, and success signal before implementation. If value cannot be measured immediately, define the learning signal that will validate or retire the bet.

## 1a. User benefit outranks the metric

Traffic, rankings, and conversions are evidence of useful work, not permission to compromise it. Follow the [SEO Constitution](./seo-constitution.md): when a metric-oriented change does not genuinely help the user, do not build it.

## 2. Performance is a feature

Fast pages create trust, improve access, and compound search performance. Prefer server rendering, static generation, responsive assets, low JavaScript cost, and progressive enhancement. Measure Core Web Vitals on meaningful page templates, not only the home page.

## 3. Accessibility is the default

Keyboard access, semantic structure, readable contrast, visible focus, reduced-motion support, and meaningful labels are baseline requirements. Accessibility work is not a cleanup phase and may not be traded away for visual novelty.

## 4. SEO serves knowledge discovery

Search optimization means making durable knowledge understandable to people and machines. Every indexable page needs a clear intent, canonical URL, useful title and description, structured data where appropriate, and internal links that reveal context.

## 5. Community before vanity metrics

Followers, likes, badges, and notifications exist to help people discover credible work and build relationships. They must not become pressure mechanisms. Measure meaningful contribution, return usefulness, and connection quality rather than raw interaction volume alone.

## 6. Knowledge before content volume

Publishing more pages is not progress if they repeat, decay, or fail to connect. Prefer well-maintained topic clusters, clear sources, editorial updates, and internal links that make the corpus more useful over time.

## 7. Identity is a shared foundation

Uretir ID is the member identity layer for every present and future product. Individual products may add scoped preferences, but they must not create fragmented accounts or duplicate consent models.

## 8. Earn trust in every state

Explain consequential actions, preserve user control, use secure defaults, and make uncertainty visible. AI recommendations must communicate that they are recommendations, not facts or professional advice.

## 9. Design systems reduce cognitive cost

Use existing patterns before creating new ones. Consistency improves speed for both users and contributors. A new component or token needs a repeatable use case, not a one-off visual need.

## 10. Think in reversible and irreversible decisions

Experiment quickly with reversible choices. Slow down for data models, URLs, identity, public APIs, privacy commitments, and infrastructure decisions. Record the latter in the decision log.
